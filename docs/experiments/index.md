# Experiments

## 目录目标

`docs/experiments/` 用于沉淀“实验层”的说明：

- 当前有哪些实验线值得维护
- 每条实验线对应哪些模型与数据集
- 推荐如何复现与比较
- 哪些实验页已经值得单独成文

它不承担 API、数据契约或架构原则说明，这些内容仍应回到 `docs/project/` 与 `docs/concepts/`。

## 当前仓库适合写成实验页的内容

结合当前实现状态，最适合优先沉淀的实验主题是：

- `itemcf` 最小闭环实验（非训练 + ranking）
- `dssm` 训练型模型实验（pointwise + binary）
- 单实验非训练路径与训练路径验证
- 多 seed 的基础 Benchmark 聚合流程

原因很简单：这些路径已经有真实代码支撑，而不是纯目录占位。

## 当前实验层的现实边界

当前需要特别注意：

- 批量 Benchmark 主干已实现，可同时包裹 itemcf（非训练）和 dssm（训练）
- 训练型 experiment 路径已接通，可通过 `dssm` 验证
- `scripts/run_single.py` 和 `scripts/run_benchmark.py` 已实现为可用 CLI 入口

## 页面类型建议

### 基线实验页

适合描述：

- 当前仓库中已经可运行的最小基线
- 输入数据要求
- 推荐评估指标
- 输出 artifact

### Benchmark 套件页

适合描述：

- 该套件跑哪些模型
- 跑哪些数据集
- 主指标是什么
- 聚合产物如何解读

### 单模型实验页

适合描述：

- 模型背景
- 当前仓库实现状态
- 所需数据字段
- 推荐配置
- 当前限制

## 推荐页面模板

每个实验页建议至少包含：

1. 页面目标
2. 当前仓库支持范围
3. 数据集与模型要求
4. 推荐配置或 API 示例
5. 评估方式
6. 输出 artifact
7. 常见失败与排查

## 文件命名规范

- 使用小写
- 使用连字符
- 让名称直接反映实验主题

例如：

- `baseline-itemcf.md`
- `benchmark-itemcf-multiseed.md`

## 当前最重要的结论

`docs/experiments/` 当前最该承接的是“真实能跑的实验线路说明”，而不是提前为尚未接通的模型家族堆放实验页面。
