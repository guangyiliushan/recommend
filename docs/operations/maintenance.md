# Maintenance Guide

## 自动化工作流

当前仓库已经包含以下 GitHub Actions 工作流：

- `ci.yml`: 静态检查、兼容性测试和覆盖率
- `deploy-docs.yml`: 文档校验与 GitHub Pages 部署
- `release.yml`: 标签触发的构建与发布

## 文档部署约定

`deploy-docs.yml` 的设计要点如下：

- 文档源目录为 `docs/`
- 站点配置文件为 `zensical.toml`
- 构建目录为 `site/`
- PR 中仅做严格构建校验
- push 与 `workflow_run` 会按变更类型判断是否部署

这意味着文档相关改动必须至少保证：

1. `docs/` 内容可构建
2. `zensical.toml` 与目录结构一致
3. 文档依赖已包含在开发环境中

## 维护优先级

建议维护顺序：

1. 修复路径与配置不一致问题
2. 稳定单实验运行主干
3. 补齐核心契约测试
4. 再增加模型覆盖范围

## 推荐维护清单

每次版本迭代前建议检查：

- README 是否仍然准确
- 文档目录与导航是否同步
- CI 路径过滤条件是否匹配当前项目结构
- benchmark 配置是否引用了真实可用的数据集与模型
- 新增模块是否有最基本的测试和文档

## 需要持续关注的风险

当前仓库最值得持续关注的风险包括：

- 目录中已有大量占位文件，但共享运行时未完成
- CI 中仍可能存在历史路径残留
- benchmark 配置与可用数据适配器并不完全一致
- 模型数量目标很大，容易先堆实现后补契约

## 文档维护要求

以下变化应同步更新文档：

- 包路径调整
- 配置结构调整
- 核心运行命令调整
- 模型接入方式调整
- CI 或发布流程调整

当仓库进入新的稳定阶段时，建议新增变更日志或版本化文档页面。
