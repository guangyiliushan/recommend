# Maintenance Guide

## 自动化工作流

当前仓库包含面向 CI、文档和发布的 GitHub Actions 工作流。

维护时最重要的不是记住每个工作流细节，而是保证下面三件事始终一致：

- 本地命令与 CI 命令一致
- 文档导航与真实文件结构一致
- README、文档与代码状态描述一致

## 文档站构建要求

当前文档站以 `docs/` 为源目录，使用 `zensical.toml` 配置。

所有文档改动都应至少保证下面命令可通过：

```bash
uv run zensical build --strict --clean
```

这也是当前最重要的文档维护基线。

## 当前维护优先级

结合仓库现状，当前建议的维护优先级是：

1. 修正文档、README、配置与代码之间的状态漂移
2. 稳定单实验与 Benchmark 主干
3. 补充公共契约测试
4. 再扩展模型覆盖范围

## 版本迭代前建议检查

每次较大改动前后，建议至少检查：

- README 是否仍准确
- `docs/` 导航与页面引用是否同步
- 配置示例是否与真实 dataclass 字段一致
- benchmark 配置是否引用真实存在的数据集与模型
- `scripts/` 是否仍被文档错误描述为稳定 CLI
- 新增模块是否补了最基本的测试或文档说明

## 当前需要持续关注的风险

- 模型目录多，但多数文件仍未实现
- 配置与脚本里可能残留旧路径或旧命名
- benchmark 配置可能引用未注册模型或数据集
- training 基础设施已存在，但 experiment 训练型路径尚未接通

## 文档维护要求

下面这些变化发生时，应同步更新文档：

- 包路径调整
- 公共 API 调整
- 配置字段调整
- 实验目录或 artifact 协议调整
- Benchmark 聚合逻辑调整
- 文档站导航或构建流程调整

## 当前最重要的结论

RecBench 当前维护工作的核心，不是“继续扩目录”，而是持续清理文档和实现之间的偏差，确保任何面对开发者的说明都真实反映当前仓库状态。
